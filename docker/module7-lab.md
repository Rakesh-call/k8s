## 🛠️ Lab Tasks & Complete Walkthrough Solution

### Phase 1: Provisioning the Persistent Storage Infrastructure

We will build a decoupled storage ecosystem using explicit, managed volume objects.

```
                  +---------------------------------------+
                  |           PHYSICAL HOST ENGINE        |
                  |                                       |
                  |   [ Volume: db-data ]                 |
                  |     Maps to: /var/lib/docker/...      |
                  |                 |                     |
                  +-----------------|---------------------+
                                    v (Mounted to target path)
                  +---------------------------------------+
                  |   Container: shipping-db (Postgres)   |
                  |     Writes directly to: /data/db      |
                  +---------------------------------------+

```

#### **Task 1.1:** Create a named managed volume destination object designated as `db-data`.

```bash
docker volume create db-data

```

* **Verification:** Run `docker volume ls` to ensure `db-data` appears in the managed storage registry.





#### **Task 1.2:** Instantiate a stateful background container named `shipping-db` using an alpine image framework. Mount your `db-data` volume directly into the container's designated data folder (`/data/db`).

```bash
docker run -d --name shipping-db --mount type=volume,source=db-data,target=/data/db alpine sh -c "while true; do echo 'Manifest ID \$(random) - Verified' >> /data/db/transactions.log; sleep 2; done"

```

#### **Task 1.3:** Verify the active real-time transaction processing. Execute an internal cat pass against the log file inside the volume layer to confirm data entry.

```bash
docker exec shipping-db cat /data/db/transactions.log

```

### Phase 2: Testing Persistence Across Catastrophic Destruction

Now, simulate an administrator accidentally deleting the container to prove that the external data volume decouples the storage layer from the container lifecycle.

#### **Task 2.1:** Forcefully terminate and permanently delete (`docker rm`) your active `shipping-db` container.

```bash
docker rm -f shipping-db

```

#### **Task 2.2:** Verify system destruction parameters. Run `docker ps -a` to confirm the container has been wiped from the engine, and then list your volumes to verify that your data survives intact.

```bash
docker ps -a
docker volume ls

```

* **Expected Output Verification:** The container is completely gone, but `db-data` remains fully registered on disk.





#### **Task 2.3:** Re-deploy a brand new database container instance using the exact same naming scheme and mount target, pointing it back to the original `db-data` volume.

```bash
docker run -d --name shipping-db --mount type=volume,source=db-data,target=/data/db alpine sh -c "while true; do echo 'Manifest ID \$(random) - Verified' >> /data/db/transactions.log; sleep 2; done"

```

#### **Task 2.4:** Read the log contents from the new container instance to verify that no past transactions were lost.

```bash
docker exec shipping-db cat /data/db/transactions.log

```

* **Expected Output Analysis:** You will see the original transaction timestamps generated by the old, deleted container alongside new log lines. The database state survived the container upgrade process completely unharmed.

### Phase 3: Implementing Volatile `tmpfs` Buffers for System Secrets

The application generates transient security session authorization keys that must never be written to any non-volatile mechanical hard drive for compliance reasons.

#### **Task 3.1:** Create a clean, separate web worker container named `auth-worker` that allocates a 32-megabyte secure **`tmpfs`** memory buffer inside RAM mapped directly into the `/var/secrets` location inside the container.

```bash
docker run -d --name auth-worker --mount type=tmpfs,destination=/var/secrets,tmpfs-size=32m alpine sh -c "while true; do echo 'SECRET_TOKEN_A93B' > /var/secrets/session.key; sleep 3600; done"

```

#### **Task 3.2:** Verify that you can read the secret key inside memory, and then check its filesystem type from the host kernel perspective.

```bash
docker exec auth-worker cat /var/secrets/session.key
docker exec auth-worker df -h /var/secrets

```

* **Expected Output Verification:** The filesystem type is explicitly registered as **`tmpfs`**, confirming that the token is safely isolated in RAM and can never leak onto physical disk sectors.

### Phase 4: Constructing a Production Volume Backup Pipeline

Since managed Docker volumes live inside highly protected system paths (`/var/lib/docker/`), normal user scripts cannot back them up directly. To fix this, you must build a classic **Data Backup Pipeline Pattern** using a temporary data-handler container.

```
  +---------------------------------------------------------------------------------+
  | HOST MACHINE                                                                    |
  |                                                                                 |
  |  [ Volume: db-data ]                          [ Target Host Path: /tmp/backup ] |
  |         |                                                      ^                |
  |         | (Mounted to /source)                                 | (Mounted to    |
  |         v                                                      |  /backup)      |
  |  +-------------------------------------------------------------+-------------+  |
  |  | Temporary Backup Container (alpine)                                        |  |
  |  | Runs: tar cvf /backup/db_export.tar /source                                |  |
  |  +---------------------------------------------------------------------------+  |
  +---------------------------------------------------------------------------------+

```

#### **Task 4.1:** Create a target host filesystem archive directory to hold your system exports.

```bash
mkdir -p /tmp/backup

```

#### **Task 4.2:** Execute the automated pipeline backup pattern. Spin up a temporary worker container that mounts your live `db-data` volume to `/source` and your host backup folder to `/backup`. Instruct it to compress the volume files into a portable tar archive, and then automatically destroy itself when finished using the `--rm` flag:

```bash
docker run --rm --mount type=volume,source=db-data,target=/source --mount type=bind,source=/tmp/backup,target=/backup alpine tar cvf /backup/db_export.tar /source

```

#### **Task 4.3:** Verify the archive footprint on your physical host filesystem.

```bash
ls -lh /tmp/backup/db_export.tar
tar -tf /tmp/backup/db_export.tar

```

* **Expected Output Verification:** A complete, compressed `db_export.tar` bundle sits safely inside your host machine's physical `/tmp/backup/` directory. This file can now be shipped securely to Amazon S3, tape drivers, or secondary offline datacenter storage systems.
